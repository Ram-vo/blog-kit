The starter now supports two runtime editorial backends:

- local filesystem content
- Supabase-backed content

The point is not to create two different products. The point is to keep
one editorial flow while letting the host choose the persistence layer.

## When local mode is the right tool

Use local mode when you want to validate:

- content structure
- the editor experience
- package boundaries
- filesystem-backed workflows

This mode is ideal when you want real editable content without external
credentials.

## When Supabase mode is the right tool

Use Supabase mode when you want to validate:

- provider-backed reads and writes
- real relational data
- a future production integration path
- Supabase Auth session mapping

This mode is where you test the adapter, not just the UI.

## The rule of thumb

If you are proving ergonomics, start local.

If you are proving infrastructure behavior, switch to Supabase.

The important design goal is that the editor surface does not change
when the backend does.
